# Security and Correctness Audit -- coinbase-trade
**Date:** 2026-03-19
**Auditor:** Claude Opus 4.6 (automated code review)
**Scope:** All core modules -- executor, engine, optimizer, risk-guard, strategies, tracker, server, MCP client, dashboard

---

## CRITICAL

### C1. No Authentication on API Endpoints -- Any Network Client Can Execute Trades
**File:** `src/web/server.ts` (all endpoints)
**Lines:** 146, 162, 383, 397, 114, 424, 525

Every HTTP endpoint -- including `POST /api/trade`, `POST /api/trade/enso`, `POST /api/settings`, `POST /api/control/pause`, `POST /api/faucet`, `POST /api/network` -- is completely unauthenticated. Any device on the same network can:
- Execute real trades (`/api/trade`)
- Disable DRY_RUN via `/api/settings` with `{ changes: { DRY_RUN: false } }`
- Pause/resume the bot
- Switch the network to mainnet
- Trigger the kill switch

**Impact:** If the dashboard port (3003) is exposed to any network (LAN, Docker bridge, public), an attacker can drain the wallet.

**Fix:** Add bearer-token or basic-auth middleware. At minimum, check a `DASHBOARD_SECRET` env var on all POST/PUT/DELETE endpoints.

---

### C2. Enso Route Accepts Arbitrary Token Addresses -- No Allowlist Validation
**File:** `src/web/server.ts:162-178`, `src/mcp/tools.ts:163-178`, `src/trading/executor.ts:113-149`

The `POST /api/trade/enso` endpoint accepts raw `tokenIn` and `tokenOut` contract addresses from the request body with no validation. Combined with C1 (no auth), an attacker can route swaps through malicious contracts.

The only guard is a mainnet-only check (`if (!botState.activeNetwork.includes('mainnet'))`), but this actually *blocks* testnet usage while allowing mainnet -- the opposite of what is safe.

**Fix:** Maintain an allowlist of approved token addresses per network. Validate `tokenIn`/`tokenOut` against it before calling `ensoRoute`.

---

### C3. `executeForAsset` Bypasses Risk Guard and Pause Check
**File:** `src/trading/executor.ts:77-111`

The `executeForAsset` method (used by all per-asset strategy loops) does **not** check:
1. Any RiskGuard veto (position limits, daily loss, portfolio floor)
2. `botState.isPaused` -- while `tickAsset` (engine.ts:126) checks isPaused, `manualTrade` (engine.ts:94-99) does NOT. A manual trade via Telegram/CLI can execute while the bot is paused.

More critically, `executeForAsset` has **zero risk guard checks** -- no position size limit, no portfolio floor, no daily loss limit. The RiskGuard is only consulted during optimizer rotations.

**Fix:** Add `isPaused` check and RiskGuard consultation to `executeForAsset`, or at minimum to `manualTrade`.

## HIGH

### H1. Rotation Leg 2 Uses Stale USDC Balance -- Race Condition
**File:** `src/trading/executor.ts:204-216`

After leg 1 (sell asset to USDC), leg 2 immediately reads `botState.lastUsdcBalance` to determine buy amount. But this balance is updated by the portfolio tracker on a polling interval -- it has NOT been refreshed after leg 1's swap. The bot uses the *pre-swap* USDC balance.

If pre-swap USDC was 0, `amount = Math.max(0 * 0.95, 0) = 0`, and leg 2 is skipped -- leaving the user in USDC with no buy-back. If pre-swap USDC was already high, the bot may buy with MORE than what leg 1 produced.

**Fix:** After leg 1 succeeds, fetch fresh USDC balance from the MCP server before proceeding to leg 2.

---

### H2. Optimizer Scores Non-ETH/USDC Assets at Price $0 -- Incorrect Rotation Sizing
**File:** `src/trading/optimizer.ts:269-272`

The sell USD value calculation falls back to `0` for any asset that is not ETH or USDC:
```
sellUsdValue = balance * (symbol === 'USDC' ? 1 : symbol === 'ETH' ? price : 0)
```

This means CBBTC, CBETH, and any discovered tokens will have `sellUsdValue = 0`, making `sellAmount = 0`. The optimizer can never rotate *out of* non-ETH assets.

**Fix:** Look up the latest price from `recentAssetSnapshots` for non-ETH/USDC assets.

---

### H3. Dashboard XSS via Server Error Messages
**File:** `src/web/public/index.html:1331-1332, 1364-1365`

Error responses from the server are injected directly into the DOM via unsafe HTML assignment. If `data.error` contains HTML (e.g., from a crafted MCP server response), it will execute in the dashboard context.

**Fix:** Use `textContent` instead of HTML assignment for error messages, or use a helper that escapes HTML entities.

---

### H4. Grid Strategy Division-by-Zero When Upper == Lower Bound
**File:** `src/strategy/grid.ts` -- `initializeLevels` method

If `upperBound` equals `lowerBound` (e.g., from zero volatility in candle data, or misconfiguration), the grid step calculation `(upper - lower) / (levels - 1)` produces `0 / N = 0`. All grid levels collapse to the same price.

If `gridLevelCount` is 1, the denominator is 0, causing `Infinity` step values.

**Fix:** Add guard: `if (this.upperBound === this.lowerBound || this.gridLevelCount < 2) return;`

---

### H5. No Database Transactions -- Concurrent Writes Can Corrupt State
**File:** `src/data/db.ts`

No SQLite transactions wrap multi-statement operations. Grid state updates (flipping `pending_buy` to `pending_sell`) are not atomic with the trade execution -- if the trade fails, the grid state is already flipped. The optimizer reads then writes daily PnL without atomicity guarantees.

**Fix:** Wrap critical multi-step operations in `db.transaction()`.

---

## MEDIUM

### M1. ThresholdStrategy Repeated Buys in Downtrend -- No Stop-Loss
**File:** `src/strategy/threshold.ts:25-44`

When a buy signal fires, `entryPrice` is reset. In a sustained downtrend, the strategy can keep buying on every tick that exceeds the drop threshold, with no stop-loss or consecutive-buy limit.

**Fix:** Consider adding a stop-loss threshold, or limiting consecutive buy signals within a time window.

---

### M2. `executeForAsset` Does Not Record Trades in DB
**File:** `src/trading/executor.ts:77-111`

Unlike `execute()` which calls `this.recordTrade()`, `executeForAsset` does not persist trades to the database or emit trade events. Per-asset trades do not appear in `/api/trades`, P&L calculations miss them, and the dashboard trade log is incomplete.

**Fix:** Add `this.recordTrade(...)` call after the swap in `executeForAsset`.

---

### M3. `executeForAsset` Error Handling -- Silent Crash Risk
**File:** `src/trading/executor.ts:108`, `src/trading/engine.ts:105`

The `swap` call at line 108 has no try/catch. The engine's asset loop at line 105 uses `void this.tickAsset(...)` -- meaning rejections from per-asset ticks may be silently swallowed or crash the process depending on the JS engine.

**Fix:** Add `.catch(err => logger.error(...))` to the setInterval callback in `startAssetLoop`.

---

### M4. Optimizer `estimatedGainPct` Is Score Delta, Not Actual Gain
**File:** `src/trading/optimizer.ts:268`

`estimatedGainPct = candidate.buy.score - candidate.sell.score` is a raw score delta (0-200 range), not a percentage. RiskGuard compares it against `estimatedFeePct` (typically 1.0%). Since score deltas are typically 40-100+, the fee gate is effectively disabled.

**Fix:** Convert score delta to an estimated percentage gain, or adjust the comparison.

---

### M5. MCP Client Has No Timeout on Tool Calls
**File:** `src/mcp/client.ts`

The `callTool` method has no timeout. If the MCP server hangs, the bot blocks indefinitely. The health check uses `fetch` with a default timeout, but `this.client.callTool()` has none.

**Fix:** Wrap `this.client.callTool()` in a `Promise.race` with a configurable timeout (e.g., 30s).

---

### M6. DRY_RUN Can Be Toggled at Runtime via Unauthenticated API
**File:** `src/web/server.ts:114-127`, `src/core/runtime-config.ts`

`DRY_RUN` is not in the `READ_ONLY_KEYS` set. Anyone with network access can disable dry-run mode via the settings API. This is a subset of C1 but worth separate attention.

**Fix:** Consider adding DRY_RUN to `READ_ONLY_KEYS`, or requiring a separate confirmation step.

---

**File:** `src/trading/engine.ts:153-154`

The engine passes `() => null` for candle data callbacks, so auto-bounds recalculation always falls back to `currentPrice * 1.05 / 0.95`, ignoring available 24h candle data.
